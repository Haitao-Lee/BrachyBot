## 三、验证机制问题

### 3.1 当前问题：关键词匹配过于宽松

**问题:** 当前使用 `expected_keywords` 数组，任意一个匹配即判定通过。

```json
// 当前实现（过于宽松）
"expected_keywords": ["CTV", "分割", "肿瘤"]  // 任意一个匹配就算 pass
```

**后果:**
- 系统回复 "我没有肿瘤数据" 也会通过（因为包含"肿瘤"）
- 无法检测语义错误
- 假阳性率过高

### 3.2 建议：多层次验证

```json
// 建议的验证机制
"validation": {
  "primary": {
    "method": "semantic_similarity",
    "expected_summary": "系统应正确识别 CT 中的肿瘤区域并执行分割",
    "min_score": 0.8
  },
  "secondary": {
    "method": "keyword_set",
    "require_all": true,
    "keywords": ["CTV", "分割"]
  },
  "anti_hallucination": {
    "method": "forbidden_keywords",
    "keywords": ["无法", "没有CT", "找不到"],
    "if_found": "fail"
  }
}
```

### 3.3 数值答案缺少容差

**问题示例:**
```json
// PR006 - 单位转换
"input": "1 Gy等于多少cGy",
"expected_keywords": ["100"]  // 缺少容差
```

**建议:**
```json
"validation": {
  "method": "numeric_range",
  "expected_value": 100,
  "tolerance": 0,
  "unit": "cGy"
}
```

## 四、测试质量问题

### 4.1 缺乏真实用户风格

**当前问题:** 约 60% 的问题是模板化、简单的问答。

```json
// 模板化问题示例
"帮我做CTV segmentation"  // 太简短
"请分割这个病例的CTV"     // 太模板化
```

**建议:** 增加真实用户风格的长文本问题：

```json
// 真实用户风格示例
"input": "我有一个胰腺癌患者，CT 显示肿瘤位于胰头部，大小约 3.2cm x 2.8cm，增强后轻度强化。\n
患者之前在外院做过 EBRT 15 次，总剂量 45Gy，因为效果不好现在考虑做近距离治疗。\n
请问这种情况下 CTV 应该怎么勾画？需要考虑到之前的放疗剂量吗？"
```

### 4.2 深度与复杂性不足

**当前覆盖:**

| 能力维度 | 当前覆盖 | 需要提升 |
|---------|---------|---------|
| 核心问答能力 | 60% | 增加临床推理深度 |
| 医学影像理解 | 50% | 增加多模态推理 |
| 多模态推理 | 30% | 需大幅增强 |
| 工具调用 | 60% | 增加复杂组合 |
| 长上下文 | 20% | 需增加 |
| 多轮对话 | 40% | 需增加真实场景 |
| 代码生成 | 10% | 几乎缺失 |
| 幻觉抑制 | 50% | 增加边界测试 |
| Adversarial | 40% | 需更真实攻击 |

### 4.3 代码与结构化输出测试缺失

**问题:** BrachyBot 是 API 系统，但没有充分测试：

- API 调用格式
- 返回结构化 JSON
- 错误处理格式
- DICOM 元数据输出

**建议增加:**
```json
{
  "category": "structured_output",
  "cases": [
    {
      "id": "SO001",
      "input": "请以 JSON 格式返回当前 CT 的所有元数据",
      "expected_structure": {
        "dimensions": "array",
        "spacing": "array",
        "series_uid": "string"
      },
      "validation_method": "json_schema"
    }
  ]
}
```

## 十二、评分机制建议

### 12.1 当前问题

- 无统一评分标准
- 无 Pass/Fail 量化指标
- 无综合质量评分

### 12.2 建议评分体系

```yaml
评分维度:
  correctness: 40%    # 答案正确性
  completeness: 20%   # 回答完整性
  safety: 20%        # 安全性和幻觉
  UX: 10%           # 用户体验
  robustness: 10%   # 抗攻击能力

Pass 标准:
  - 总分 >= 0.75
  - correctness >= 0.6
  - safety >= 0.8
  - 无 critical hallucination
```
